# canopy v0.0.1 — first public release

**MIT-licensed Python treedocs clone.** This is the initial public release — every subcommand except `fill` is fully functional, byte-compatible with upstream DandyLyons/treedocs YAML format, and tested.

## What's in this release

### CLI subcommands

- `canopy init` — create an empty `canopy.yaml` if missing
- `canopy check` — exit 1 on drift or missing descriptions (CI-friendly)
- `canopy show [--depth N]` — print the tree with descriptions
- `canopy explore` — compact one-line-per-path view (agent-friendly)
- `canopy update PATH DESC` — set one description
- `canopy fill` — **stub in this release**; Phase 1 lands it in v0.1.0

### On-disk format

- Byte-compatible with upstream `DandyLyons/treedocs` (same schema URL `0.2.0`, same `_kind` / `_description` keys, same `tree` / `signature` top-level shape)
- SHA256 signature over sorted paths; descriptions intentionally NOT part of the signature (per upstream behavior)

### What works against the upstream tool

If you have an existing `treedocs.yaml` from the Swift tool, `canopy` reads and writes it without migration.

## Install

```sh
pip install -e ".[dev]"
canopy --version
```

## Verify it works

```sh
canopy init
canopy update README.md "top-level readme"
canopy check     # OK
canopy show
```

## What's NOT in this release (and why)

- **`fill` subcommand** — wired through `cli.py` but returns "not yet implemented (Phase 1)". The bws + MiniMax LLM loop is the only piece left; we're landing it in the next minor release. If you need fills now, use `~/.hermes/scripts/canopy.py fill` (the source-drop) or run `canopy update PATH DESC` manually.
- **PyPI package** — not published yet. For now: `git clone` + `pip install -e .`.
- **Homebrew tap** — no plans unless usage justifies it.
